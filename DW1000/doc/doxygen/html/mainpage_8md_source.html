6 RIOT in a nutshell {#riot-in-a-nutshell}
7 ==================
8 RIOT is an open-source microkernel-based operating system, designed to match
9 the requirements of Internet of Things (IoT) devices and other embedded
10 devices. These requirements include a very low memory footprint (on the order
11 of a few kilobytes), high energy efficiency, real-time capabilities,
12 communication stacks for both wireless and wired networks, and support for a
13 wide range of low-power hardware.
14 
15 RIOT provides a microkernel, multiple network stacks, and utilities which
16 include cryptographic libraries, data structures (bloom filters, hash tables,
17 priority queues), a shell and more. RIOT supports a wide range of
18 microcontroller architectures, radio drivers, sensors, and configurations for
19 entire platforms, e.g. Atmel SAM R21 Xplained Pro, Zolertia Z1, STM32 Discovery
20 Boards etc. (see the list of
21 [supported hardware](https://github.com/RIOT-OS/RIOT/wiki/RIOT-Platforms).
22 Across all supported hardware (32-bit, 16-bit, and 8-bit platforms). RIOT
23 provides a consistent API and enables ANSI C and C++ application programming,
24 with multithreading, IPC, system timers, mutexes etc.

27 Structure {#structure}
28 =========
29 
30 This section walks you through RIOT's structure. Once you understand this
31 structure, you will easily find your way around in RIOT's code base.
32 
33 ![Overview](riot-structure.svg)
34 
35 RIOT's code base is structured into five groups.
36 
37  - The kernel (`core`)
38  - Platform specific code (`cpu`; `boards`)
39  - Device drivers (`drivers`)
40  - Libraries and network code (`sys`; `pkg`)
41  - Applications for demonstrating features and for testing (`examples`;
42  `tests`)
43 
44 In addition RIOT includes a collection of scripts for various tasks (`dist`) as
45 well as a predefined environment for generating this documentation (`doc`).
46 
47 The structural groups are projected onto the directory structure of RIOT, where
48 each of these groups resides in one or two directories in the main RIOT
49 directory.
50 
51 The following list gives a more detailed description of each of RIOT's
52 top-level directories:
53 
54 core
55 ----
56 This directory contains the actual kernel. The kernel consists of the
57 scheduler, inter-process-communication (messaging), threading, thread
58 synchronization, and supporting data-structures and type definitions.
59 
60 See @ref core for further information and API documentations.
61 
62 boards
63 ------
64 The platform dependent code is split into two logic elements: CPUs and boards,
65 while maintaining a strict 1-to-n relationship, a board has exactly one CPU,
66 while a CPU can be part of n boards. The CPU part contains all generic, CPU
67 specific code (see below).
68 
69 The board part contains the specific configuration for the CPU it contains.
70 This configuration mainly includes the peripheral configuration and
71 pin-mapping, the configuration of on-board devices, and the CPU's clock
72 configuration.
73 
74 On top of the source and header files needed for each board, this directory
75 additionally may include some script and configuration files needed for
76 interfacing with the board. These are typically custom flash/debug scripts or
77 e.g. OpenOCD configuration files. For most boards, these files are located in a
78 `dist` sub-directory of the board.
79 
80 See here @ref boards for further information.
81 
82 cpu
83 ---
84 For each supported CPU this directory contains a sub-directory with the name of
85 the CPU. These directories then contain all CPU specific configurations, such
86 as implementations of power management (LPM), interrupt handling and vectors,
87 startup code, clock initialization code and thread handling (e.g. context
88 switching) code. For most CPUs you will also find the linker scripts in the
89 `ldscripts` sub-directory.
90 
91 In the `periph` sub-directory of each CPU you can find the implementations of
92 the CPU's peripheral drivers like SPI, UART, GPIO, etc. See @ref drivers_periph
93 for their API documentation.
94 
95 Many CPUs share a certain amount of their code (e.g. all ARM Cortex-M based
96 CPUs share the same code for task switching and interrupt handling). This
97 shared code is put in its own directories, following a `xxxxx_common` naming
98 scheme. Examples for this is code shared across architectures (e.g.
99 `cortexm_common`, `msp430_comon`) or code shared among vendors (e.g.
100 `kinetis_common`).
101 
102 See @ref cpu for more detailed informtation.
103 
104 drivers
105 -------
106 This directory contains the drivers for external devices such as network
107 interfaces, sensors and actuators. Each device driver is put into its own
108 sub-directory with the name of that device.
109 
110 All of RIOT's device drivers are based on the peripheral driver API (e.g. SPI,
111 GPIO, etc.) and other RIOT modules like the `xtimer`. This way the drivers are
112 completely platform agnostic and they don't have any dependencies into the CPU
113 and board code.
114 
115 See @ref drivers for more details.
116 
117 sys
118 ---
119 RIOT follows the micro-kernel design paradigm where everything is supposed to
120 be a module. All of these modules that are not part of the hardware abstraction
121 nor device drivers can be found in this directory. The libraries include data
122 structures (e.g. bloom, color), crypto libraries (e.g. hashes, AES) ,
123 high-level APIs (e.g. Posix implementations), memory management (e.g. malloc),
124 the RIOT shell and many more.
125 
126 See @ref sys for a complete list of available libraries
127 
128 sys/net
129 -------
130 The `sys/net` sub-directory needs to be explicitly mentioned, as this is where
131 all the networking code in RIOT resides. Here you can find the network stack
132 implementations (e.g. the @ref net_gnrc "GNRC" stack) as well as network stack agnostic code as
133 header definitions or network types.
134 
135 See @ref net for more details on networking code.
136 
137 pkg
138 ---
139 RIOT comes with support for a number of external libraries (e.g.
140 [ccn-lite](https://github.com/cn-uofbasel/ccn-lite),
141 [microcoap](https://github.com/1248/microcoap)). The way they are included is
142 that RIOT ships with a custom Makefile for each supported library that
143 downloads the library and optionally applies a number of patches to make it
144 work with RIOT. These Makefiles and patches can be found in the `pkg`
145 directory.
146 
147 See @ref pkg for a detailed description on how this works.
148 
149 examples
150 --------
151 Here you find a number of example applications that demonstrate certain
152 features of RIOT. The default example found in this directory is a good
153 starting point for anyone who is new to RIOT.
154 
155 For more information best browse that directory and have a look at the
156 `README.md` files that ship with each example.
157 
158 To create your own application — here or anywhere else — see @ref creating-an-application
159 
160 tests
161 -----
162 Many features/modules in RIOT come with their own test application, which are
163 located in this directory. In contrary to the examples these tests are mostly
164 focusing on a single aspect than on a set of features. Despite for testing, you
165 might consider these tests also for insights on understanding RIOT.
166 
167 dist & doc
168 ----------
169 All the tooling around RIOT can be found in these two folders.
170 
171 `doc` contains the doxygen configuration and also contains the compiled doxygen
172 output after running `make doc`.
173 
174 Lastly, the `dist` directory contains tools to help you with RIOT. These
175 include
176 the serial terminal application `pyterm`, generic scripts for flashing,
177 debugging, reseting (e.g. support for [OpenOCD](http://openocd.org/),
178 [Jlink](https://www.segger.com/jlink_base.html)), as well as code enabling easy
179 integration to open testbeds such as the [IoT-LAB](https://www.iot-lab.info/).
180 Furthermore you can find here scripts to do all kind of code and style checks.
